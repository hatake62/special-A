## スライド別メモ

### 1. Special-A

これは、Python風のソースコードを自作コンパイラでVM命令に変換し、自作VMで実行する処理系です。単に動くだけではなく、最適化して実行命令数、時間、消費エネルギーを下げるところまで作りました。

### 2. 何を作ったか

最低ラインである関数と配列に対応しています。関数は `def`、引数、`return`、呼び出しがあります。配列はリテラル、添字アクセス、添字代入、`len` に対応しています。さらに、while、if、四則演算、try/except/raise も動きます。

### 3. 書けるプログラム

この例では、配列を受け取る関数 `sum_array` を定義し、while で配列を走査しています。Python風に見えますが、Pythonで直接実行しているのではなく、内部で独自ASTに変換し、自作VM命令として実行しています。

### 4. 処理の流れ

流れは、ソース、構文解析、最適化、コンパイル、VM実行です。構文解析だけは Python 標準の `ast` を使っていますが、その後の内部AST、VM命令、実行器、最適化はこのプロジェクトで実装しています。

### 5. スタックで実行する

VMはスタックマシンです。例えば `10 + 20` は、10を積み、20を積み、`add` 命令で2つを取り出して足します。関数呼び出しではフレームを積み、配列は `build_array`、`get_index`、`set_index` で扱います。命令数を数えているので、最適化の効果を数値で説明できます。

### 6. デモ1

実行コマンド:

```bash
python3 main.py examples/array.src
```

期待される出力:

```text
10
25
3
65
```

ここでは、配列の読み出し、更新、長さ取得、関数に配列を渡した合計計算を見せます。

### 7. デモ2

実行コマンド:

```bash
python3 main.py examples/demo.src
```

期待される出力:

```text
777
20
```

0除算や `raise` が起きたときに、VMが例外ハンドラへ制御を移します。普通の関数と配列だけでなく、例外処理までVMで扱える点を示します。

### 8. コンパイル時に計算を消す

定数畳み込みは、実行しなくても答えが分かる式をコンパイル時に計算してしまう最適化です。例えば `(10 * 20 + 30 * 40 - 50)` は常に `1350` なので、ループのたびに計算する必要がありません。

### 9. 小さい関数呼び出しを消す

関数インライン展開は、単純な `return` だけの関数呼び出しを、その関数本体の式に置き換える最適化です。関数呼び出しのフレーム作成や `call` / `ret` のぶんを減らせます。

### 10. 命令数が減る

測定では、定数畳み込み・配列のケースで 1,165,020 命令から 845,020 命令に減りました。インライン展開でも 540,020 命令から 460,020 命令に減っています。ここでは実行時間だけでなく、VM命令数という環境に左右されにくい指標を見ています。

### 11. 消費エネルギーも下がる

Kubernetes 上でベンチマークPodを動かし、Kepler と Prometheus で消費電力を測りました。`combined_array` では、最適化なしと比べて消費エネルギーを約37.0%削減できました。平均消費電力だけでなく、実行時間を含めたエネルギーで見ることが重要です。

### 12. デモ3

実行コマンド:

```bash
python3 bench/run_case.py combined_array --compare --repeat 1
```

見るべき点:

```text
result_match: true
no_opt instruction_count: 680020
opt instruction_count:    460020
```

結果が同じまま、VM命令数が減っていることを示します。実行時間はその場の負荷で揺れるので、発表では命令数を中心に説明します。

### 13. まとめ

凄い点は3つです。1つ目は、関数・配列・制御構文・例外を持つ言語を実装したこと。2つ目は、スタック、フレーム、例外テーブルを持つVMを作ったこと。3つ目は、最適化を実装して、実測で命令数・時間・エネルギーの改善を示したことです。

## 質疑で答えやすい補足

- なぜ Python の `ast` を使ったか: パーサを一から作るより、VM・最適化・測定に集中するため。
- Pythonそのものとの違い: Python互換処理系ではなく、Python風構文を独自ASTと自作VMで動かす小さな言語。
- 最適化しても安全か: `run_case.py --compare` で最適化前後の結果一致を確認している。
- 実行時間が毎回変わる理由: OS負荷などで揺れるため。VM命令数と同一環境での相対比較を重視する。
- 省エネ測定の考え方: 平均消費電力だけではなく、`消費エネルギー = 平均消費電力 × 実行時間` で比較する。
