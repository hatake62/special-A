## 発表用考察

今回の測定では、定数畳み込みと関数インライン展開のどちらも、最適化によって実行時間と VM 命令数が減少し、消費エネルギーも削減された。

定数畳み込みでは、実行時間が `108.7091 sec` から `73.8674 sec` へ短縮され、VM 命令数も `116502000` から `84502000` へ減少した。これは、実行時に毎回評価していた定数式をコンパイル時に計算済みの値へ置き換えたことで、ループ内で繰り返し実行される命令を大きく削減できたためである。命令数の削減が実行時間の短縮につながり、結果として消費エネルギーも約34.4%削減できた。

関数インライン展開でも、実行時間は `65.1754 sec` から `47.8358 sec` に短縮され、VM 命令数は `54002000` から `46002000` に減少した。関数呼び出しに伴う処理を減らせたことで、実行時間とエネルギー消費の削減につながった。一方で、平均消費電力は最適化後の方が高くなっている。しかし、実行時間の短縮効果が大きかったため、総消費エネルギーは約13.9%削減された。

この結果から、電力効率を評価するときは平均消費電力だけを見るのではなく、実行時間を含めた消費エネルギーで比較することが重要だと分かる。特に、最適化によって短時間で処理を終えられる場合、瞬間的な消費電力が同程度または増加しても、最終的なエネルギー消費は小さくなる可能性がある。

また、定数畳み込みの方がインライン展開よりもエネルギー削減率が大きかった。これは、今回のベンチマークでは定数式が繰り返し実行されるホットパスにあり、削減した命令数の効果が積み重なったためである。最適化の効果は、単に最適化を適用できるかどうかだけでなく、削減対象の処理がどれだけ頻繁に実行されるかに強く依存する。

## combined_array の段階的な測定結果

`combined_array` ベンチマークについて、最適化なし、定数畳み込みのみ、定数畳み込みとインライン展開を組み合わせた場合の測定結果をまとめる。

- benchmark: `combined_array`
- repeat: `100`
- 測定環境: Kubernetes 上の Pod、Kepler/Prometheus、`zone=package`
- 消費エネルギー: `average_power_w × elapsed_sec` で計算

| 条件 | 実行時間 | VM命令数 | 平均消費電力 | 消費エネルギー |
| --- | ---: | ---: | ---: | ---: |
| 最適化なし | 61.61369603500498 sec | 68002000 | 2.548751350401555e-06 W | 0.00015703799097244986 J |
| 定数畳み込みのみ | 52.58650619299442 sec | 52002000 | 2.3973908452256558e-06 W | 0.00012607040852948709 J |
| 定数畳み込み + インライン展開 | 40.329941827003495 sec | 46002000 | 2.454071646816674e-06 W | 0.00009897256675541511 J |

### combined_array の改善率

| 条件 | 実行時間 | VM命令数 | 消費エネルギー |
| --- | ---: | ---: | ---: |
| 定数畳み込みのみ | 約14.7%短縮 | 約23.5%削減 | 約19.7%削減 |
| 定数畳み込み + インライン展開 | - | - | 定数畳み込みのみと比べて約21.5%削減、最適化なしと比べて約37.0%削減 |

### combined_array の考察

定数畳み込みにより、実行時に評価していた定数式の計算命令を削減できた。その結果、最適化なしと比べて実行時間は約14.7%短縮され、VM 命令数は約23.5%削減され、消費エネルギーも約19.7%削減された。

さらにインライン展開を組み合わせることで、関数呼び出しに伴う `call` / `ret` を削減できた。消費エネルギーは定数畳み込みのみの場合と比べてさらに約21.5%削減され、最適化なしと比べると約37.0%削減された。

平均消費電力は必ずしも常に下がるわけではないが、実行時間の短縮により総消費エネルギーを削減できた。Kepler/Prometheus による Kubernetes Pod 上の測定では、絶対値そのものよりも、同一環境・同一条件での相対比較を重視する。
